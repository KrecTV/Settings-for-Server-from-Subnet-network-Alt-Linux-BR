Сервер филиала (BR-SRV) – мини-копия HQ-SRV

Филиал — это не просто удалённая точка, а полноценный боевой пост, которому нужен собственный сервер. Да, он заметно проще головного собрата: здесь нет DNS-роли и всего один сетевой сегмент. Но задачи те же — предоставить администратору безопасный удалённый доступ и быть надёжным бастионом на случай, если туннель в центр по какой-то причине моргнёт. Настраиваем почти зеркально, но с поправкой на локальные адреса.

### 5.1. Имя и интерфейс

Даём машине уникальное имя, чтобы при входе сразу понимать, где мы находимся.

	hostnamectl hostname srv-br.example.com
	exec bash

`hostnamectl` прописывает все варианты имени (статическое, динамическое и красивое), а `exec bash` перезапускает оболочку, подхватывая новое приглашение без лишних телодвижений. Теперь консоль говорит нам, что мы именно на сервере филиала.

В отличие от головного офиса, у нас нет возни с VLAN: сервер подключён в единственную локальную сеть филиала 192.168.3.0/29. Поэтому достаточно одного простого Ethernet-интерфейса. Создадим под него каталог конфигурации.

	cd /etc/net/ifaces
	mkdir enp0s3

Сразу определяем базовые опции: это обычный провод, адрес зададим статически, интерфейс должен подниматься при загрузке и после рестарта сети.

	echo -e "TYPE=eth\nBOOTPROTO=static\nONBOOT=yes" > enp0s3/options

Теперь конкретные сетевые параметры. Сервер получает адрес 192.168.3.2/29 — второй (и фактически единственный клиентский) адрес в этой маленькой подсети. Шлюзом по умолчанию, разумеется, выступает локальный маршрутизатор филиала BR-RTR с адресом 192.168.3.1.

	echo 192.168.3.2/29 > enp0s3/ipv4address
	echo default via 192.168.3.1 > enp0s3/ipv4route

Пока собственный DNS не готов (а его тут и не будет), временно направляем разрешение имён на публичный сервер Google. Потом мы это исправим, когда центральный DNS войдёт в строй.

	echo nameserver 8.8.8.8 > enp0s3/resolv.conf
	systemctl restart network

После перезапуска сетевой службы сервер уже пингует свой шлюз и может обновляться.

### 5.2. Пользователь sshuser

Для единообразия во всей инфраструктуре заводим того же самого пользователя `sshuser` с фиксированным UID 2026. Это важно: если мы когда-нибудь будем переносить файлы или бекапы, права не превратятся в тыкву из-за расхождения идентификаторов.

	useradd -u 2026 -m sshuser
	passwd sshuser
	usermod -aG wheel sshuser
	echo "sshuser ALL=(ALL:ALL) NOPASSWD:ALL" >> /etc/sudoers

Сначала создаём учётку с домашним каталогом и конкретным UID, задаём пароль (вводим дважды надёжную фразу), затем добавляем её в группу `wheel` — это даёт право использовать `sudo`. Последней строкой дописываем в `/etc/sudoers` разрешение на выполнение любых команд без повторного ввода пароля. Удобно для удалённого администрирования и скриптов.

### 5.3. SSH-сервер

Теперь превратим сервер в крепость. Установим OpenSSH-сервер (в минимальной установке ALT Linux его могло не быть) и заодно nano — вдруг кто-то предпочитает его для правки конфигов.

	apt-get update
	apt-get install -y nano openssh-server

Главный файл настроек демона sshd лежит по стандартному пути. Открываем его.

	nano /etc/openssh/sshd_config

Внутри нужно раскомментировать или добавить несколько ключевых директив:

Port 2026
AllowUsers sshuser
MaxAuthTries 2
Banner /etc/openssh/banner.txt

После сохранения и выхода осталось создать сам баннер — текстовое предупреждение, которое увидит каждый подключающийся ещё до приглашения аутентификации.

	echo "Authorized access only" > /etc/openssh/banner.txt
	systemctl enable --now sshd

Теперь sshd слушает нестандартный порт 2026, принимает соединения только от пользователя `sshuser`, даёт всего две попытки на ввод пароля и показывает грозное предупреждение. Боты-сканеры, привыкшие к порту 22, останутся ни с чем.

### 5.4. Часовой пояс

Время должно быть точным и единым во всей сети. Без этого логи рассыпаются, а в будущем могут возникнуть проблемы с Kerberos или истечением сертификатов.

	apt-get install tzdata -y
	timedatectl set-timezone Europe/Moscow
	timedatectl status

Пакет `tzdata` поставляет информацию о часовых поясах. `timedatectl` переключает систему на московское время, а финальный `status` выводит сводку, чтобы можно было глазами убедиться в успехе.

### 5.5. Переключение DNS

До этого мы использовали 8.8.8.8, но настоящий резолвинг внутренних имён возможен только через центральный DNS-сервер головного офиса (192.168.10.2). Как только он готов — переключаемся на него. Тогда имена вроде `srv-hq.example.com` или `web.example.com` будут разрешаться корректно.

	cd /etc/net/ifaces
	echo nameserver 192.168.10.2 > enp0s3/resolv.conf
	systemctl restart network

Сначала мы зашли в каталог, чтобы не писать длинные пути, затем перезаписали файл `resolv.conf` новым адресом и применили изменения через рестарт сети. Теперь филиал полностью влился во внутреннюю DNS-зону.

Сервер филиала оперился и готов принимать SSH-подключения. На нём можно спокойно работать удалённо, зная, что доступ защищён, время синхронизировано, а имена резолвятся через центральный узел. Филиал больше не оторванный остров, а полноценная часть корпоративной сети.
